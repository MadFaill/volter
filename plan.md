## Принятые решения (зафиксировано)

1. **Единый стек — Rust + React.** Backend, оркестратор и движок качества — на **Rust** в одной
   кодовой базе. Frontend — **React + TypeScript + Vite + Tailwind**. Опыт `/opt/cply-agent` (движок
   на TS) берём как **архитектурную модель** (фазы/роли/гейты/event-sourcing), но **пишем заново на
   Rust** — единый язык в проде, общие типы control-plane ↔ движок, единый CI/деплой.
2. **Без монетизации — только cost/time-учёт.** Из экономической модели «нейроцеха» берём идею
   **учёта реальной стоимости и времени** каждого ответа LLM (event-sourcing-проекция `costRollup`),
   агрегаты по диалогу/проекту/связке agent+model. **НЕ** строим: пакеты ⚡, СБП/ЮKassa, маржу,
   авто-рефилл, шаринг энергии, биллинг-gateway. Volter — личный инструмент, не SaaS на продажу.

## Часть I. Принципы построения и архитектура

### 1. Как соотносятся README и прототипы

- **README = ЧТО строим** (критичные требования, объём, поведение). Любой шаг трассируется в требование
  README (см. §17).
- **Прототипы = КАК уже пробовали строить** (архитектурные решения, что сработало, где грабли).
  Из них берём подтверждённые паттерны и анти-паттерны (§2, §3). Не берём их код и не повторяем их
  ограничения слепо.

### 2. Проверенные подходы из прототипов, которые принимаем за основу

> Это дизайн-решения, доказавшие себя в предыдущих итерациях. Реализуем их заново в `volter`.

**Из `/opt/volt` (control-plane и инфра):**
- **Dialog-first SDLC**: идея → план (read-only артефакт) → «В работу» (freeze плана + создание run) →
  execution → checks → test-deploy → ручной prod-deploy. Поведение, которое стоит сохранить.
- **Двусторонний git**: internal bare-origin (управляется системой, work-ветки) + external (GitHub) с
  mirror-синхронизацией; ssh-ключ генерится на проект. Хорошая модель изоляции и пушей.
- **Артефакт-гейт (безмодельная валидация)**: каждый шаг плана несёт `check.cmd` + `expect_contains/absent`,
  который система прогоняет сама — не доверяя «assert» от LLM. Берём как второй контур поверх TDD-гейтов.
- **Runtime-plane как абстракция исполнения**: подготовка workspace, прогон стадий, сбор артефактов —
  отдельный слой. Именно его HTTP/gRPC-режим даёт распределённость (в прототипе был только local).
- **Серийность исполнения на проект** через execution-lock с лизами — защита от гонок.
- **Изоляция управляемых продуктов**: рабочая нагрузка не имеет доступа к секретам/исходникам системы.
- **Test/prod-контуры** раздельными compose-проектами, backup перед prod, rollback, OOM-защита билдов,
  nginx wildcard-роутинг проектных доменов. Грабли уже пройдены (напр. инцидент «run снёс prod-БД» →
  жёсткая изоляция тестовой БД).
- **Cost/time на уровне диалога и run**, аналитика срезами.

**Из `/opt/cply-agent` (движок качества «нейроцех»):**
- **Конвейер фаз SDLC** (intake → triage → research → requirements → approval → test_modeling →
  test_writing → development → build → verification → deploy_stand → e2e_stand → release_verdict →
  promotion → post_mortem). Это и есть требуемый README «основной SDLC + шаги по задаче».
- **Микроцикл фазы plan→synth→gate (shift-left)**: дорогой план (что реально надо) → дешёвый synth
  (собрать артефакт) → детерминированный/verdict-гейт; провал = self-loop с feedback, не дальний откат.
- **Роли с границами (separation of powers)**: роль, меняющая код, ≠ роль, меняющая правила; намерение
  (манифест) неприкосновенно, команда двигает только реальность (код).
- **TDD-гейты, доказывающие реальность тестов**: `test-modeled` → `tests-written` (red) → `tests-green`
  (с baseline-diff от merge-base) + **`test-honesty`** (тест проверяет do/verify, а не заглушку) +
  `contract-coverage`. Прямо закрывает требование README «тесты реальные, не mock».
- **Event-sourcing**: `events`-лог рана — единственный источник истины; состояние/бюджеты/стоимость/
  resume/stuck-detection — чистые проекции лога. Даёт детерминизм, replay и golden-тесты.
- **Классификация задачи** (triage + estimate-size S/M/L) → разный профиль шагов плана.
- **Author-планирование**: research → product (корень) → перспективы (architecture/security/ux/…) →
  единый канон `main.md` → нарезаемые манифесты с адресуемыми unit'ами и `codeAnchors`.
- **Git meta-sync по шагам**: коммит на завершение шага, пуш на ключевые переходы, delivery-ledger
  с курсорами и replay недоставленного.
- **Провижининг нод**: per-проект OS-user, изоляция, systemd-таймеры опроса — модель для «сетап по кнопке».
- **Адаптеры раннеров за единым интерфейсом** (claude-code/codex/aider/shell/docker-shell/stub) —
  модель абстракции агента и изоляции исполнения в контейнере.

### 3. Чего в прототипах нет (или сделано лишь частично), а README требует — проектируем заново

1. **Реальная распределённость**: в `/opt/volt` runtime-plane был только local; нужны воркеры на
   отдельных EU-VPS, управление с RU-ноды, межхостовая связь через Docker overlay + mTLS.
2. **Provisioning ноды «по кнопке» из UI** control-plane под её тип (worker/stand-test/stand-prod),
   с супервизором прогона скриптов, созданием юзеров на VPS.
3. **Враппер авторизации агентов из web** (claude login / codex auth / aider api-key) — UI-флоу без LLM.
4. **Единый стек Rust для движка** (в прототипе движок был на TS) — проектируем фазы/роли/гейты на Rust.
5. **UI**: Tailwind (был кастомный CSS), голосовой ввод, WS/SSE-стримы вместо polling, трансляция
   «мыслей» агента, прогресс задачи в стиле deep research, дерево/просмотр файлов проекта с подсветкой.
6. **Research-флоу первого класса**: запустить исследование → «сохранить в git» кнопкой → обсудить → план.
7. **DB-ассистент и infra-ассистент** внутри проекта.
8. **Расширяемые skills с генерацией агентом** (в прототипе skills детерминированы, без саморасширения).
9. **Память проекта**: индекс с тегами для быстрого recall, осмысленные авто-названия диалогов,
   живая автодока — довести до полноценного.

### 4. Целевая архитектура (новая система)

```
┌──────────────────────────── CONTROL PLANE (RU-VPS) — ставится вручную ──────────────────────┐
│  React/Vite/Tailwind UI ──HTTP/WS──► API (Rust/Axum) ──► Orchestrator+Worker (Rust)           │
│  Postgres │ secrets-store │ git internal/external │ node-manager │ cost/time-analytics         │
└───────────────────────────────────────────┬───────────────────────────────────────────────────┘
                          runtime-plane (gRPC/HTTP over mTLS, Docker overlay)
        ┌──────────────────────────┬─────────────────────────┬──────────────────────────┐
        ▼                          ▼                         ▼                          ▼
 ┌────────────────┐       ┌────────────────┐        ┌────────────────┐        ┌────────────────┐
 │ WORKER (EU)    │       │ WORKER (EU)    │        │ STAND test     │        │ STAND prod     │
 │ volter-engine  │       │ volter-engine  │        │ деплой проекта │        │ деплой проекта │
 │ (Rust) +       │       │ (Rust) +       │        │ + e2e-прогон   │        │ (app+db compose)│
 │ claude-code CLI│       │ codex / aider  │        │                │        │                │
 └────────────────┘       └────────────────┘        └────────────────┘        └────────────────┘
```

Слои (концепция three-layers из «нейроцеха»): **Control (RU)** — управление/UI/git/аналитика;
**Worker pool (EU)** — исполнение агентов и движка качества; **Deployment (test/prod)** — стенды проектов.

> **Фаза 1 (сейчас): всё на одной ноде** `volter.comalert.pw` (control-plane + worker + агенты
> co-located), runtime-plane в local-режиме. Это целевая архитектура «свёрнутая в одну машину»;
> разнесение на EU-воркеры (Ш3–Ш5) — позже, через тот же интерфейс runtime-plane, без переписывания.
> Детали развёртывания и логина — §11.

### 5. Стек (фиксируем)

- Control-plane backend/оркестратор/движок: **Rust** (axum, tokio, sqlx, Postgres) — единый крейт-воркспейс
  (`control-plane`, `engine`, `runtime-plane`, общие `types`).
- Frontend: **React + TypeScript + Vite + Tailwind CSS**. Тема — **светлая, минималистичная**;
  диалог и контролы по образцу **ChatGPT** (мелкие инлайновые переключатели модели/режима). Mobile-first,
  iPad, extra-small. Тёмная «приборная» эстетика прототипа VOLT не используется (см. Ш10, §19).
- Транспорт CP↔ноды: **gRPC/HTTP over mTLS** в Docker overlay; UI↔CP: **WebSocket/SSE** для стримов.
- Хранилища: Postgres (метаданные), файловое хранилище (`.agent/` md-память, артефакты), secrets-store.
- Всё через Docker; межхостовая связь — через Docker overlay.

## 6. Контрактная YAML-модель: правила, критерии и нарезка шагов (ядро системы)

> Это центральное архитектурное решение. Оба прототипа независимо пришли к идее: **«что делать» и
> «как это проверить» выражаются декларативно**, а исполнение — это прогон против машинно-проверяемых
> критериев. Берём лучшее из обоих и делаем единую модель из **трёх слоёв YAML**.

### 6.1. Идея в одной фразе
**Манифест и роли — это стоячие правила и критерии проекта (канон, «что такое хорошо»). План —
нарезка ИМЕННО этой задачи на шаги с контрактом `do`/`verify`/`check`. Исполнение — прогон шагов
против безмодельных проверок и гейтов ролей; провал возвращает feedback в следующую попытку.**

```
СЛОЙ 1: КАНОН ПРОЕКТА (стоячие правила/критерии, YAML в .agent/)   ← редко меняется, неприкосновенен
   manifest units (правила: арх/безопасность/ux/вординг/регуляторика) + roles (правила+критерии)
        │  задают рамку и критерии качества; роль владеет своими гейтами
        ▼
СЛОЙ 2: ПЛАН ЗАДАЧИ (нарезка под конкретную задачу, генерится на диалог)   ← VOLT-подход
   stages[].steps[] = { do, verify, check{cmd,expect}, satisfies:[unit.id] }
        │  детерминированные правила нарезки (kind, обяз. tests/root_cause, синтез валидатора)
        ▼
СЛОЙ 3: ИСПОЛНЕНИЕ + ГЕЙТ-ПЕТЛЯ (event-sourcing)
   шаг → безмодельный check + гейты ролей → GateDecision{status,signal,feedback}
        провал → feedback инжектится в промпт следующей попытки (бюджет ретраев, human-in-the-loop)
        успех → коммит шага, handoff следующему
```

### 6.2. Слой 1 — Канон: правила и критерии как YAML (из «нейроцеха», но декларативно)

**Manifest unit = адресуемое правило/критерий.** Живёт в `.agent/manifest/<section>.yaml`
(секции: architecture / security / ux / wording / data / regulatory / product …). Канон
неприкосновенен для исполняющих ролей (separation of powers).

```yaml
# .agent/manifest/architecture.yaml
units:
  - id: ARCH.LAYER.ddd                 # стабильный адрес (не меняется по цепочке)
    section: architecture
    severity: hard                     # hard|soft — блокирует или предупреждает
    tags: [layering, ddd]              # fine-отбор релевантного среза в промпт
    text: "Бэкенд по DDD: domain/application/infrastructure/interfaces; зависимости только внутрь."
    codeAnchors: ["backend/src/domain", "backend/src/interfaces"]   # мост к реальному коду
    anchorsTo: [PRODUCT.QUALITY.maintainable]                       # трассировка к корню PRODUCT.*
```

**Роль = правила (`guard`) + критерии (`gate`) в YAML** — это улучшение над прототипом, где роль была
TS-кодом с `buildPrompt`. Делаем роли **декларативными**: тело промпта — шаблон, в который движок
инжектит контекст и feedback; код движку не нужен.

```yaml
# .agent/roles/architect.yaml
id: architect
purpose: "Привязать каждое требование к реальным символам кода."
manifestSections: [architecture, core]   # какой срез канона видит роль (coarse)
codeAccess: anchored                      # search | anchored — как получает код
guard:
  does: "Привязывает каждое требование к реальным символам/модулям."
  mustNot: "Менять anchor, состав или продуктовый смысл требований."
gate: [symbol-contract, product-anchored] # КРИТЕРИИ, которые обязана пройти роль
promptTemplate: roles/architect.md        # шаблон; движок добавит guard/код/манифест/feedback
```

### 6.3. Слой 2 — План: нарезка шагов под задачу (из VOLT, ведущий подход)

Планировщик режет **именно текущую задачу** на стадии и шаги. Каждый шаг несёт контракт
**`do` (что сделать) + `verify` (критерий словами) + `check` (машинная проверка)** и `satisfies`
(каким правилам канона удовлетворяет).

```yaml
# .agent/dialogs/<id>/plan-1.yml — нарезка ПОД ЭТУ задачу
class: feature                 # из triage; задаёт скелет стадий и набор обяз. шагов
stages:
  - id: S1
    kind: tests                # для feature/impl tests-стадия ОБЯЗАТЕЛЬНА (детерм. правило)
    files: [backend/src/domain/teaser_test.rs]
    steps:
      - do: "написать red-тест teaser_returns_by_uuid"
        verify: "тест компилируется и падает (red) до реализации"
        check: { cmd: "cd backend && cargo test teaser_returns_by_uuid", expect_contains: "test result" }
        satisfies: [PRODUCT.SCENARIO.public_teaser]
    done: "Контракт теста существует и красный до реализации."
  - id: S2
    kind: implementation
    files: [backend/src/domain/teaser.rs]
    steps:
      - do: "реализовать Teaser::by_uuid"
        verify: "тест teaser_returns_by_uuid зелёный"
        check: { cmd: "cd backend && cargo test teaser_returns_by_uuid", expect_contains: "ok",
                 requires_validator: false }
        satisfies: [PRODUCT.SCENARIO.public_teaser]
    done: "Сценарий проходит реальным тестом."
```

**Детерминированные правила нарезки (проверяются БЕЗ LLM; невалидный план отклоняется):**
- `kind ∈ {implementation, refactor, scaffold, tests, validation, root_cause, e2e, docs, deploy}`.
- Каждый шаг обязан иметь непустые `do`, `verify` и `check` (либо `cmd`, либо `requires_validator+validator_file`).
- Есть `implementation/refactor/scaffold` → **обязательна** `tests`-стадия.
- Класс `bug` → первая стадия `root_cause`.
- `implementation`-стадии обязаны называть физические `files`.
- `requires_validator: true` → движок **сам вставляет отдельную `tests`-стадию**, которая физически
  пишет валидатор, прежде чем исходная проверка может стать зелёной (из VOLT — закрывает «нечем проверить»).
- Эвристика объёма: класс `meta/контент` с одним файлом → короткий **rewrite-flow** (1–2 шага);
  код → полный TDD-flow (tests → impl → e2e).

### 6.4. Слой 3 — Исполнение и гейт-петля (event-sourcing)

- План **замораживается** при «В работу» (`frozen`), исполняется по шагам строго по очереди.
- Каждый шаг: выполнила роль → **безмодельный `check`** (passed / failed / `missing_validator`
  при «0 тестов» / skipped при отсутствии инструмента; таймаут) **+ гейты ролей** (det/verdict).
- Результат — `GateDecision { status, signal, feedback[] }`; `feedback` = `{code, where, problem, fix}`.
- Провал → по `signal` маршрутизатор решает: self-loop (та же фаза) или возврат к нужной роли;
  **feedback инжектится в промпт следующей попытки** («исправь, иначе тот же возврат»); бюджет
  ретраев на канал; при исчерпании или критичном сигнале — **human-in-the-loop**.
- Успех шага → коммит в ветку диалога, `handoff` (контекст) следующему шагу.
- Всё пишется в `events`-лог рана; стоимость/время/прогресс/resume/stuck — **чистые проекции лога**.

### 6.5. Связки между слоями
- `step.satisfies[] → manifest.unit.id` — шаг доказывает конкретное правило канона; гейт
  `contract-coverage` проверяет, что **каждое затронутое правило закрыто** хотя бы одним зелёным шагом.
- `manifest.unit.anchorsTo → PRODUCT.*` — трассировка к продуктовому корню (BFS); гейт отклоняет
  «осиротевшие» требования вне продуктовой рамки.
- `task_class → {скелет стадий, набор manifestSections, набор ролей}` — класс задачи выбирает, какие
  правила и роли применяются и как режется план.
- Роль владеет своими `gate`; терминальные гейты фазы — на уровне фазы.

### 6.6. Почему так (а не иначе)
- **Безмодельные `check` + декларативные критерии** не дают LLM «самозачесть» работу — закрывает
  требование README «тесты реальные, не mock» структурно, а не на доверии.
- **Нарезка под задачу (VOLT)** даёт гибкость «середины» SDLC: план = базовый каркас + шаги под
  конкретную задачу, как требует README, без жёсткого зашитого пайплайна.
- **Правила/критерии в YAML (нейроцех)** делают качество (архитектура/ui/вординг/регуляторика)
  явным, версионируемым и переиспользуемым между задачами; роль как YAML — проще расширять, чем код.
- **Feedback-петля** превращает провал гейта в прицельную доработку, а не в слепой ретрай.

## 7. Связки агент+модель: модель, привязка к действиям, fallback (фундамент)

> Связка `agent+model` — **базовая единица исполнения**: любое действие LLM выполняется какой-то
> связкой. Ключевые вопросы, которые здесь закрываем: **что такое связка**, **к каким действиям и где
> она привязывается**, **в каком порядке выбирается** (приоритет переопределений) и **как
> деградирует (fallback)**, если что-то недоступно.

### 7.1. Что такое связка (Binding)
```yaml
# элемент каталога связок проекта (.agent/bindings.yaml → bindings:)
- id: claude-sonnet                 # короткий ключ для UI и роутинга
  agent: claude-code                # claude-code | claude | codex | aider | shell
  model: claude-sonnet-4-6          # конкретная модель (для shell/безмодельных — пусто)
  tier: medium                      # fast | medium | qualified — для tier-резолва и fallback
  mode_params: { reasoning: medium } # под-режим рассуждения/параметры
  auth_ref: secrets://nodes/eu-1/claude  # где авторизована (Ш6); определяет доступность
  fallback: [codex-gpt5, deepseek-reasoner]  # цепочка деградации (ссылки на id связок)
```
- `agent` бывает **агентный** (claude-code/codex/aider — пишет код, гоняет тесты в cwd проекта),
  **текстовый** (claude/codex как трансформер) и **безмодельный** (shell — для verification/e2e).
- Доступность связки определяется авторизацией агента на ноде (Ш6) + лимитами модели.

### 7.2. К каким действиям привязывается связка (action/phase → binding)
Связка привязывается **к действию (фазе)**, а не «к диалогу вообще». Каждая фаза SDLC и каждая роль
имеют свою связку. Это и есть ответ на «где задаётся связка для планирования, исследования,
уточнения, разработки, написания тестов».

```yaml
# .agent/bindings.yaml — профиль связок проекта (роутинг действий)
defaults:
  chat:          claude-haiku        # диалог/уточнение — быстро и дёшево
  research:      claude-sonnet       # исследование (web-discovery)
  planning:      claude-opus         # план: дорого, «что реально надо»
  test_modeling: claude-sonnet       # контракт теста
  test_writing:  claude-sonnet       # red-тесты
  development:   claude-code-sonnet  # разработка — АГЕНТНАЯ связка (реальный код+тесты)
  review:        claude-haiku
  verification:  shell               # безмодельно
  e2e:           shell

role_pins:                           # точечный пин роли (перебивает phase default)
  developer:
    require_kind: agentic            # «жёсткий» пин: разработчик обязан быть агентным
    prefer: claude-code-sonnet       # «мягкий» выбор модели (можно переопределить)

fallback_policy:
  triggers: [unauthorized, unavailable, rate_limited, budget_low, node_down]
  threshold: { min_remaining_fraction: 0.05 }  # <5% остатка лимита → переключение
  preserve: tier                     # при фолбэке держим тот же tier через каталог модели
```

### 7.3. Где задаётся связка (3 уровня + приоритет разрешения)
Связка для действия разрешается по приоритету (сверху вниз — побеждает первое заданное):

1. **Per-message override** — пользователь выбрал связку в композере для конкретной отправки (разово).
2. **Dialog override** — выбранная в композере связка «прилипает» к диалогу для текущего режима
   (chat/planning/execution) и его действий.
3. **Role pin** — `role_pins` (напр. `developer` обязан быть агентным — `require_kind` не переопределяется
   пользователем; `prefer`-модель — может).
4. **Project default** — матрица `defaults` (action→binding) в `.agent/bindings.yaml`.
5. **System default** — встроенные дефолты, если проект ничего не задал.

После разрешения применяется **fallback** (§7.4), если выбранная связка недоступна.

### 7.4. Fallback связок (деградация при недоступности)
- **Триггеры:** агент не авторизован/недоступен на ноде, модель `rate_limited` (429), остаток лимита
  < порога, нода упала.
- **Алгоритм:** идём по `fallback`-цепочке связки; для каждой следующей резолвим модель **того же
  tier** из каталога (`preserve: tier`) — т.е. `qualified`-план не падает на `fast`. Если вся цепочка
  недоступна → human-in-the-loop («нет доступной связки для planning — авторизуйте агента / выберите
  другую»).
- **Прозрачность:** какая связка реально отработала и почему сработал fallback — пишется в event-лог
  рана и показывается в UI бейджем (`planning: codex-gpt5 ← fallback от claude-opus (rate_limited)`).
- **Безмодельные действия** (verification/e2e на `shell`) fallback не имеют — они детерминированы.

### 7.5. Связь с авторизацией и нодами
- Связка «доступна», только если её `agent` авторизован (Ш6) на ноде нужного типа/региона и модель
  в лимите. Каталог связок в UI показывает статус каждой (✓auth / нет авторизации / 429 / нода down).
- Воркер-нода под EU может нести разные агенты (claude на одной, codex на другой) — роутер выбирает
  ноду по доступности связки.

### 8.2. Переключатель режима = выбор «как работаем»
Сегментед-контрол `Чат · План · Иссл. · Разработка` задаёт текущий **режим диалога**, а режим
определяет, **какое действие** (и какая связка по §7) сработает на отправку:
- **💬 Чат** → действие `chat` (уточнение/обсуждение), связка `defaults.chat`.
- **🔬 Иссл.** → действие `research` (запуск исследования, прогресс как deep research), связка `research`.
- **📋 План** → действие `planning` (формирование/правка плана), связка `planning`.
- **⚙ Разработка** → запуск исполнения замороженного плана: **много фаз, каждая со своей связкой**
  (test_modeling/test_writing/development/review/verification…). Здесь композер-связка показывает не
  одну модель, а **профиль** (см. 8.4).

### 8.3. Инлайн-выбор связки (chat / research / planning — одно действие)
Компактный dropdown у композера (как «модель» в ChatGPT), раскрывается вверх:

- Выбор тут = **dialog override** для текущего режима (§7.3, ур.2); галочка «дефолт проекта» пишет в

## 9. Память проекта: сквозная и локальная, мета-индекс, безмодельный поиск

> README требует память проекта с индексом по тегам и мета-контекстом, чтобы агент быстро понимал
> «о чём речь», **не загружая всё подряд**. Это критическая подсистема. Делаем её как
> **детерминированный retrieval** (RAG/MCP-стиль, но без LLM в самом поиске): LLM лишь формулирует
> запрос («сейчас поищу про X»), а **ищет код** (Rust + unix-инструменты) и возвращает контекст структурно.

### 9.1. Два контура памяти
- **Локальная (по диалогу):** полный тред диалога + рабочий набор файлов/артефактов этого диалога.
  `.agent/dialogs/<id>/`. Держит контекст беседы и созданное внутри диалога.
- **Сквозная (по проекту):** агрегат знаний по всему проекту — решения, обсуждения, исследования,
  живая архитектура. `.agent/memory/`. Чтобы новый диалог поднимал релевантное из прошлого.

### 9.2. Структура на диске (контекст проекта = клон git)
```
.agent/
  memory/
    index.db          # SQLite: теги, FTS по summary/title, обратные ссылки — быстрый поиск
    index.md          # человекочитаемый индекс (строка на карточку) — версионируется в git
    cards/<id>.md     # карточки памяти: одна тема/решение, frontmatter (теги/summary/файлы/ссылки)
  dialogs/<id>/
    thread.md         # полный диалог (локальная память)
    summary.md        # сжатая выжимка → попадает в сквозной индекс
    artifacts/        # созданное в диалоге (коммитится по мере создания, §10)
    plan-*.yml
  architecture.md     # живая автодока (индексируется как карточка type=architecture)
```

### 9.3. Карточка памяти (атом сквозной памяти)
```yaml
---
id: mem-2026-06-27-teaser-uuid
type: decision           # decision | dialog | research | architecture | fix | contract
tags: [teaser, uuid, portal, public-api]
summary: "Публичная выдача teaser по uuid без авторизации: решение и контракт."
files: [backend/src/domain/teaser.rs]
dialog: dlg-5882df56
commit: a30efe0
date: 2026-06-27
links: [PRODUCT.SCENARIO.public_teaser, mem-2026-06-20-portal-routing]
---
Что обсуждалось, что решили и почему. (тело — для подгрузки по требованию)
```
Карточки связываются `links` между собой и с `manifest.unit.id`/`PRODUCT.*` — мост к контрактной модели §6.

### 9.4. Безмодельный retrieval (ядро)
LLM **не грузит всё**. Цикл:
1. LLM формулирует **намерение поиска** («нужен контекст про nginx-роутинг и teaser») — это всё, что она делает.
2. **Retriever (Rust, без LLM)** ищет детерминированно по нескольким каналам:
   - **тег-индекс** (`index.db`) — точное совпадение по тегам;
   - **FTS** по `summary`/`title` карточек и выжимок диалогов;
   - **ripgrep / git grep** по телам карточек, докам и **по коду проекта**;
   - **git log --grep / blame** по истории изменений («вектор изменений» из README).
3. Возврат **структурно**: ранжированный список `{id, type, tags, summary, file_refs, snippet}` —
   компактно, без полных тел.
4. LLM по списку решает, что подгрузить целиком → `memory.get(id)` / `code.read(path:range)`.

Принцип: **поиск контекста — без LLM**; LLM только говорит «что искать» и получает структурный результат.
Это «RAG, но лексический/структурный» — без эмбеддингов на старте (эмбеддинги — опциональный канал позже).

### 9.5. Инструменты агента (детерминированные; MCP/skill-поверхность)
- `memory.search(query?, tags?, scope=project|dialog, limit)` → ранжированные карточки/выжимки.
- `memory.get(id)` → полное тело карточки/дока.
- `code.search(pattern, path?)` → ripgrep по клону проекта.
- `code.read(path, range?)` → срез файла.
- `history.search(query)` → git log/grep/blame.
Все — на Rust + unix (rg/fd/git), **без LLM**. Агент «больше ищет инструментами», а не «вспоминает по памяти модели».

### 9.6. Индексация (когда обновляется)
- При **фиксации диалога** и на ключевых шагах (§10): извлечь теги/summary/файлы → записать карточку
  + строку в `index.md` + запись в `index.db`. Извлечение тегов/summary — дешёвой моделью **один раз на
  фиксацию**; сам последующий поиск — без LLM.
- `architecture.md` переиндексируется при изменении (живая автодока).
- `index.md` версионируется в git → история знаний прослеживается тегами.

## 10. Git и сохранность артефактов диалога

> Проблема (из брифа): агент в диалоге создаёт много артефактов, но любая манипуляция с git
> (checkout/branch/reset) затрёт незакоммиченное. Нужно сохранять (коммитить) и пушить **по мере
> создания**, гранулярно, не теряя ни артефакты, ни контекст диалога.

### 10.1. Изоляция: ветка и worktree на диалог (РЕШЕНО)
**Зафиксированный выбор: один git worktree на диалог.** Каждый **диалог = своя git-ветка +
отдельный `git worktree`** (рабочая копия). Альтернатива «один общий worktree + commit перед
переключением» — **отклонена**: она хрупка (любой незакоммиченный артефакт при переключении
диалога/рана теряется или конфликтует) и сериализует работу. Worktree-на-диалог даёт настоящую
изоляцию и параллельность.

- Переключение между диалогами **не делает `checkout`** в общей копии → артефакты одного диалога
  физически не затираются другим.
- **Дёшево по диску:** worktree’ы делят **одну объектную базу** `.git` (дублируется только рабочее
  дерево файлов, не история). На крупных репозиториях это приемлемо.
- **Жизненный цикл:** worktree создаётся при первом изменении в диалоге (lazy), а не на каждый чат;
  чисто-разговорные диалоги без артефактов worktree не заводят. Неактивные worktree’ы **архивируются/
  убираются** (`git worktree prune` + `git worktree remove`) по политике (напр. диалог завершён/вмёржен
  или N дней без активности); ветка и память остаются в истории.
- **Лимиты:** мягкий кап на число одновременных worktree’ов на проект; при превышении — самые старые
  неактивные сворачиваются (ветка сохраняется, рабочее дерево материализуется заново по запросу).
- Полноценные раны (разработка) исполняются в **отдельном runtime-workspace** (как VOLT: `run-<id>`),
  результат вливается в ветку диалога. Прод-данные/секреты в workspace недоступны (изоляция VOLT,
  грабли «run снёс prod-БД» — не повторяем).

### 10.2. Гранулярный коммит «по мере создания»
- Артефакт создан/изменён → **сразу коммит** (по одному файлу/логической единице) в ветку диалога,
  затем **push** в origin. «Висящего» незакоммиченного состояния, которое можно потерять, не остаётся.
- Контекст диалога (`thread.md`, `summary.md`, карточки памяти, индекс §9) коммитится тем же механизмом —
  диалог и его память версионируются вместе с кодом.
- «Зафиксировать» в UI (§8.7) — явная точка: commit+push текущего состояния диалога и обновление индекса.

### 10.3. Безопасные git-операции
- Все git-операции движка — **аддитивные на ветке диалога**; деструктивные (`reset --hard`, `clean`,
  `checkout` поверх изменений) запрещены, пока есть незакоммиченное (сначала commit).
- Слияние в основную ветку — только на промоушене (как VOLT, merge перед prod), не в ходе диалога.
- Теги на ключевых вехах → индекс «вектора изменений» (§9.6).

  бейджи статусов/fallback, разворачиваемая панель прогресса — всё extra-small.
- Лёгкий sidebar со списком диалогов (как ChatGPT/Claude); на мобиле — бургер.
- Mobile-first и iPad: те же мелкие контролы хорошо ложатся на тач; адаптация под 320px и планшет.
- **DoD:** UI-кит на Tailwind в светлой минималистичной теме реализует макеты §8; диалог визуально и по
  контролам близок к ChatGPT; проходит на 320px и iPad; визуальная регрессия зафиксирована.

#### Шаг 11 — Диалоговый движок: режимы, выбор связок, стрим мыслей, deep-research-прогресс
> Поведение и привязка связок к действиям — §7; раскладка контролов — §8.
- **WebSocket/SSE** вместо polling.
- **Переключатель режима** (segmented `Чат · План · Иссл. · Разработка`, §8.2): режим определяет
  действие и связку по §7.
- **Инлайн-выбор связки в композере (§8.3):** для одно-действийных режимов (chat/research/planning) —
  compact dropdown agent+model+под-режим = **dialog override** (§7.3); галочка «дефолт проекта» пишет
  в матрицу. Недоступные связки показаны disabled с причиной.
- **Режим «Разработка» (§8.4):** инлайн-контрол показывает **профиль связок** (пресеты
  проектный/быстрый/качество) и фазовую матрицу, т.к. фаз много и у каждой своя связка.
- **Прогресс задачи как deep research (§8.6):** этапы=фазы, под-шаги, живой апдейт, **реально
  отработавшая связка на фазе + бейдж fallback**; стрим «мыслей» в простых диалогах.
- **Мета-строка** (§8.7): git-ветка, стенд, agent+model, статус тестов, cost/time.
- **DoD:** в диалоге мелкими контролами переключаются режим и связка (как в ChatGPT); в «Разработке»
  виден профиль связок и пофазный прогресс с fallback-бейджами; тред чистый и светлый.

#### Шаг 12 — Память проекта (сквозная+локальная), мета-индекс, безмодельный retrieval (§9, §10)
- **Два контура памяти (§9.1):** локальная (`.agent/dialogs/<id>/`) и сквозная (`.agent/memory/`
  с `cards/`, `index.md`, `index.db`).
- **Карточки памяти (§9.3)** с frontmatter (теги/summary/файлы/links→`PRODUCT.*`/`manifest.unit`);
  `architecture.md` как живая автодока.
- **Безмодельный retriever (§9.4–9.5)** на Rust + unix (ripgrep/git/SQLite-FTS): инструменты
  `memory.search` / `memory.get` / `code.search` / `code.read` / `history.search`. LLM формулирует
  запрос — поиск исполняется детерминированно, результат отдаётся структурно.
- **Индексация (§9.6)** при фиксации диалога: извлечь теги/summary дешёвой моделью один раз, дальше — без LLM.
- **Git-сохранность (§10):** ветка+worktree на диалог; гранулярный commit+push «по мере создания»;
  деструктивные git-операции запрещены при незакоммиченном.
- **Осмысленные авто-названия** диалогов; список как в ChatGPT/Claude с **cost+time**.
- **DoD:** новый диалог создаёт ветку и worktree; артефакты коммитятся по мере создания (не теряются);
  после фиксации в `index.md`/`index.db` появляется карточка с тегами; в новом диалоге `memory.search`
  по тегам без LLM находит релевантные карточки и агент подгружает только их (контекст не раздувается).

#### Шаг 13 — Просмотр файлов проекта и аналитика
- Дерево файлов проекта + превью с подсветкой (read-only по умолчанию, README п.41).
- Аналитика: срезы по проекту, по всем проектам, по агенту и **связке agent+model** (cost/time/успех тестов).
- **DoD:** любой файл проекта открывается с подсветкой; дашборд показывает срезы по agent+model.

#### Шаг 14 — Голосовой ввод
- Контрол микрофона → STT → текст агенту (провайдер STT — §18).
- **DoD:** голосовое сообщение распознаётся и обрабатывается агентом в диалоге.

### Этап E. Research, ассистенты, скилы, стоимость

#### Шаг 15 — Research-флоу первого класса
- Сценарий: обсудить → **запустить исследование** (роль researcher, web-discovery) → прогресс как
  deep research → **«сохранить в git» кнопкой** (`.agent/research/<id>.md`) → обсудить → план → «В работу».
- **DoD:** исследование запускается из диалога, результат коммитится в ветку и доступен в плане.

#### Шаг 16 — Project env, DB-ассистент, infra-ассистент
- Project env с доступами (БД и т.д.), секреты в secrets-store (не в git).
- **DB-ассистент**: аналитика/запросы внутри проекта (напр. база tecdoc) через project env.
- **Infra-ассистент**: быстрые правки инфры (nginx и т.п.) в контексте проекта.
- **DoD:** DB-ассистент выполняет аналитический запрос; infra-ассистент правит nginx через диалог.

#### Шаг 17 — Расширяемые skills с генерацией агентом
- Реестр скилов, расширяемый из проекта; путь, когда **агент сам создаёт скил** по запросу
  (генерация + регистрация + тест скила) поверх детерминированной базовой модели.
- **DoD:** по запросу «нужен инструмент X» агент создаёт скил, он регистрируется и используется в фазе.

#### Шаг 18 — Точная стоимость и время (без монетизации)
- Реализовать cost/time-учёт как **проекцию event-лога** (`CostEntry`/`costRollup`): стоимость каждого
  ответа LLM и время его работы, агрегаты по диалогу/проекту/связке agent+model. Цены моделей — из
  конфигурации (прайс per-1M токенов).
- **Не** реализуем: energy/CAP, платежи, пакеты, gateway-биллинг (решение №2).
- **DoD:** для каждого ответа агента видны cost и time; `costRollup` сходится в тестах; платёжных
  сущностей в схеме нет.

### Этап F. Деплой, стенды, релиз

#### Шаг 19 — Деплой проектов на test/prod и e2e на стенде
- Фазы `deploy_stand`/`e2e_stand`/`promotion`: деплой проекта на стенды (compose, rollback, backup,
  nginx wildcard-роутинг), полноценные e2e на test-стенде, prod — только по явному подтверждению.
- **DoD:** проект едет идея→план→разработка→test(e2e зелёные)→prod по кнопке; rollback работает.

#### Шаг 20 — Развёртывание control-plane и сквозная приёмка
- Один скрипт/кнопка для разворачивания **только CP** (RU-VPS); дальше всё из UI (Ш5/Ш6).
- Сквозной приёмочный e2e: поднять CP → добавить EU-воркер кнопкой → авторизовать агента из web →
  добавить git-проект → диалог → research → план → разработка с TDD → test-стенд → prod.
- Финальная автодока, тег релиза.
- **DoD:** сценарий README §62 («поднять инфру, агентов на разных VPS, CP — и работать») проходит
  целиком одним сквозным тестом.
