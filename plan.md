# Volter — детальный план реализации

> **Что мы делаем:** строим **новую систему `volter` с нуля** в этом репозитории.
> **Источник истины по требованиям — `README.md`** (критичные требования: что именно должно быть).
>
> **Роль прототипов `/opt/volt` и `/opt/cply-agent` — банк опыта, не код для переноса.**
> Это две предыдущие итерации той же идеи. Мы изучили их и **извлекли проверенные архитектурные
> подходы и грабли**, чтобы новая система строилась на этом опыте, а не с чистого листа концептуально.
> - `/opt/volt` — рабочий control-plane (Rust/Axum + React/Vite + Postgres + Docker): диалоги, планы,
>   runs, фазы, test/prod-деплой, runtime-plane, артефакт-гейт, git internal/external + ssh на проект,
>   аналитика cost/time.
> - `/opt/cply-agent` («нейроцех») — TS-движок оркестрации: 15 фаз SDLC, роли, валидаторы/TDD-гейты,
>   микроцикл plan→synth→gate (shift-left), event-sourcing, манифесты, author-планирование,
>   provisioning, LLM-экономика.
>
> ⛔ **Табу:** `/opt/volt` и `/opt/cply-agent` **не редактируем, не импортируем, не мигрируем их код**.
> Только читаем как референс архитектуры. Весь код пишем заново в `volter`.
>
> Этот документ — **последовательный набор шагов**. Пройдя их все, получаем систему из README.
> Каждый шаг: цель → задачи → какой опыт прототипов берём → Definition of Done (DoD).
>
> Принцип №0 (README п.77): **делать качественно с первого раза.** Каждый шаг сам закрывается
> реальными (не mock) тестами, автодокой и фиксацией в git с тегами/индексом.

## Принятые решения (зафиксировано)

1. **Единый стек — Rust + React.** Backend, оркестратор и движок качества — на **Rust** в одной
   кодовой базе. Frontend — **React + TypeScript + Vite + Tailwind**. Опыт `/opt/cply-agent` (движок
   на TS) берём как **архитектурную модель** (фазы/роли/гейты/event-sourcing), но **пишем заново на
   Rust** — единый язык в проде, общие типы control-plane ↔ движок, единый CI/деплой.
2. **Без монетизации — только cost/time-учёт.** Из экономической модели «нейроцеха» берём идею
   **учёта реальной стоимости и времени** каждого ответа LLM (event-sourcing-проекция `costRollup`),
   агрегаты по диалогу/проекту/связке agent+model. **НЕ** строим: пакеты ⚡, СБП/ЮKassa, маржу,
   авто-рефилл, шаринг энергии, биллинг-gateway. Volter — личный инструмент, не SaaS на продажу.

## Часть I. Принципы построения и архитектура

### 1. Как соотносятся README и прототипы

- **README = ЧТО строим** (критичные требования, объём, поведение). Любой шаг трассируется в требование
  README (см. §17).
- **Прототипы = КАК уже пробовали строить** (архитектурные решения, что сработало, где грабли).
  Из них берём подтверждённые паттерны и анти-паттерны (§2, §3). Не берём их код и не повторяем их
  ограничения слепо.

### 2. Проверенные подходы из прототипов, которые принимаем за основу

> Это дизайн-решения, доказавшие себя в предыдущих итерациях. Реализуем их заново в `volter`.

**Из `/opt/volt` (control-plane и инфра):**
- **Dialog-first SDLC**: идея → план (read-only артефакт) → «В работу» (freeze плана + создание run) →
  execution → checks → test-deploy → ручной prod-deploy. Поведение, которое стоит сохранить.
- **Двусторонний git**: internal bare-origin (управляется системой, work-ветки) + external (GitHub) с
  mirror-синхронизацией; ssh-ключ генерится на проект. Хорошая модель изоляции и пушей.
- **Артефакт-гейт (безмодельная валидация)**: каждый шаг плана несёт `check.cmd` + `expect_contains/absent`,
  который система прогоняет сама — не доверяя «assert» от LLM. Берём как второй контур поверх TDD-гейтов.
- **Runtime-plane как абстракция исполнения**: подготовка workspace, прогон стадий, сбор артефактов —
  отдельный слой. Именно его HTTP/gRPC-режим даёт распределённость (в прототипе был только local).
- **Серийность исполнения на проект** через execution-lock с лизами — защита от гонок.
- **Изоляция управляемых продуктов**: рабочая нагрузка не имеет доступа к секретам/исходникам системы.
- **Test/prod-контуры** раздельными compose-проектами, backup перед prod, rollback, OOM-защита билдов,
  nginx wildcard-роутинг проектных доменов. Грабли уже пройдены (напр. инцидент «run снёс prod-БД» →
  жёсткая изоляция тестовой БД).
- **Cost/time на уровне диалога и run**, аналитика срезами.

**Из `/opt/cply-agent` (движок качества «нейроцех»):**
- **Конвейер фаз SDLC** (intake → triage → research → requirements → approval → test_modeling →
  test_writing → development → build → verification → deploy_stand → e2e_stand → release_verdict →
  promotion → post_mortem). Это и есть требуемый README «основной SDLC + шаги по задаче».
- **Микроцикл фазы plan→synth→gate (shift-left)**: дорогой план (что реально надо) → дешёвый synth
  (собрать артефакт) → детерминированный/verdict-гейт; провал = self-loop с feedback, не дальний откат.
- **Роли с границами (separation of powers)**: роль, меняющая код, ≠ роль, меняющая правила; намерение
  (манифест) неприкосновенно, команда двигает только реальность (код).
- **TDD-гейты, доказывающие реальность тестов**: `test-modeled` → `tests-written` (red) → `tests-green`
  (с baseline-diff от merge-base) + **`test-honesty`** (тест проверяет do/verify, а не заглушку) +
  `contract-coverage`. Прямо закрывает требование README «тесты реальные, не mock».
- **Event-sourcing**: `events`-лог рана — единственный источник истины; состояние/бюджеты/стоимость/
  resume/stuck-detection — чистые проекции лога. Даёт детерминизм, replay и golden-тесты.
- **Классификация задачи** (triage + estimate-size S/M/L) → разный профиль шагов плана.
- **Author-планирование**: research → product (корень) → перспективы (architecture/security/ux/…) →
  единый канон `main.md` → нарезаемые манифесты с адресуемыми unit'ами и `codeAnchors`.
- **Git meta-sync по шагам**: коммит на завершение шага, пуш на ключевые переходы, delivery-ledger
  с курсорами и replay недоставленного.
- **Провижининг нод**: per-проект OS-user, изоляция, systemd-таймеры опроса — модель для «сетап по кнопке».
- **Адаптеры раннеров за единым интерфейсом** (claude-code/codex/aider/shell/docker-shell/stub) —
  модель абстракции агента и изоляции исполнения в контейнере.

### 3. Чего в прототипах нет (или сделано лишь частично), а README требует — проектируем заново

1. **Реальная распределённость**: в `/opt/volt` runtime-plane был только local; нужны воркеры на
   отдельных EU-VPS, управление с RU-ноды, межхостовая связь через Docker overlay + mTLS.
2. **Provisioning ноды «по кнопке» из UI** control-plane под её тип (worker/stand-test/stand-prod),
   с супервизором прогона скриптов, созданием юзеров на VPS.
3. **Враппер авторизации агентов из web** (claude login / codex auth / aider api-key) — UI-флоу без LLM.
4. **Единый стек Rust для движка** (в прототипе движок был на TS) — проектируем фазы/роли/гейты на Rust.
5. **UI**: Tailwind (был кастомный CSS), голосовой ввод, WS/SSE-стримы вместо polling, трансляция
   «мыслей» агента, прогресс задачи в стиле deep research, дерево/просмотр файлов проекта с подсветкой.
6. **Research-флоу первого класса**: запустить исследование → «сохранить в git» кнопкой → обсудить → план.
7. **DB-ассистент и infra-ассистент** внутри проекта.
8. **Расширяемые skills с генерацией агентом** (в прототипе skills детерминированы, без саморасширения).
9. **Память проекта**: индекс с тегами для быстрого recall, осмысленные авто-названия диалогов,
   живая автодока — довести до полноценного.

### 4. Целевая архитектура (новая система)

```
┌──────────────────────────── CONTROL PLANE (RU-VPS) — ставится вручную ──────────────────────┐
│  React/Vite/Tailwind UI ──HTTP/WS──► API (Rust/Axum) ──► Orchestrator+Worker (Rust)           │
│  Postgres │ secrets-store │ git internal/external │ node-manager │ cost/time-analytics         │
└───────────────────────────────────────────┬───────────────────────────────────────────────────┘
                          runtime-plane (gRPC/HTTP over mTLS, Docker overlay)
        ┌──────────────────────────┬─────────────────────────┬──────────────────────────┐
        ▼                          ▼                         ▼                          ▼
 ┌────────────────┐       ┌────────────────┐        ┌────────────────┐        ┌────────────────┐
 │ WORKER (EU)    │       │ WORKER (EU)    │        │ STAND test     │        │ STAND prod     │
 │ volter-engine  │       │ volter-engine  │        │ деплой проекта │        │ деплой проекта │
 │ (Rust) +       │       │ (Rust) +       │        │ + e2e-прогон   │        │ (app+db compose)│
 │ claude-code CLI│       │ codex / aider  │        │                │        │                │
 └────────────────┘       └────────────────┘        └────────────────┘        └────────────────┘
```

Слои (концепция three-layers из «нейроцеха»): **Control (RU)** — управление/UI/git/аналитика;
**Worker pool (EU)** — исполнение агентов и движка качества; **Deployment (test/prod)** — стенды проектов.

### 5. Стек (фиксируем)

- Control-plane backend/оркестратор/движок: **Rust** (axum, tokio, sqlx, Postgres) — единый крейт-воркспейс
  (`control-plane`, `engine`, `runtime-plane`, общие `types`).
- Frontend: **React + TypeScript + Vite + Tailwind CSS** (mobile-first, iPad, extra-small).
- Транспорт CP↔ноды: **gRPC/HTTP over mTLS** в Docker overlay; UI↔CP: **WebSocket/SSE** для стримов.
- Хранилища: Postgres (метаданные), файловое хранилище (`.agent/` md-память, артефакты), secrets-store.
- Всё через Docker; межхостовая связь — через Docker overlay.

## Часть II. Последовательные шаги реализации

> Шаги — порядок получения работающей системы. Каждый шаг закрывается реальными тестами, автодокой,
> коммитами по шагам, тегом и записью в индекс памяти.

### Этап A. Каркас новой системы

#### Шаг 0 — Bootstrap репозитория `volter` (с нуля)
- Rust-воркспейс: крейты `control-plane` (API+оркестратор), `engine` (движок качества), `runtime-plane`
  (абстракция исполнения), `shared-types`.
- Frontend-каркас: React + Vite + Tailwind.
- Docker: `docker-compose.yml` (dev), заготовки `*.test.yml`/`*.prod.yml`; `Makefile`/`justfile`.
- CI с TDD-гейтом (fmt, clippy, тесты backend; typecheck/build/тесты frontend).
- **DoD:** `volter` собирается, поднимается локально (пустой каркас), CI зелёный. Код прототипов не
  импортирован — только конспект их подходов лежит в `docs/prototype-lessons.md`.

#### Шаг 1 — Доменная модель и миграции
- Спроектировать схему под README + подходы §2: `projects`, `dialogs`, `messages`, `plans`, `runs`,
  `events` (event-sourcing рана), `jobs`, `nodes`, `bindings` (agent+model+mode), `deployments`,
  `phases/roles/gates`, `task_class`, `manifest_units`, `cost_entries`.
- Зафиксировать инварианты: `dialog`→git-ветка, `run`→event-log, `plan`→`main.md`/манифесты.
- **DoD:** миграции применяются, HTTP-тесты роутинга зелёные, схема описана в `docs/data-model.md`.

#### Шаг 2 — Абстракция исполнения (runtime-plane) и единый интерфейс агента
- Спроектировать трейт исполнения: `prepare_workspace` / `run_stage` / `collect_artifacts` /
  `stream_logs` / `stream_thoughts`; local-реализация для разработки.
- Единый интерфейс агента (по опыту адаптеров «нейроцеха»): `claude-code` / `codex` / `aider` /
  `shell` / `docker-shell`; изоляция исполнения в контейнере; cwd = только код проекта.
- **DoD:** локальный прогон фиктивной стадии через runtime-plane; интерфейс агента покрыт тестами.

### Этап B. Распределённость и ноды

#### Шаг 3 — Runtime-plane HTTP/gRPC: реальный воркер-протокол
- Реализовать удалённый режим runtime-plane поверх mTLS: задания на воркер, стрим логов/мыслей,
  возврат артефактов и коммитов на CP. Серийность на проект через execution-lock.
- Воркер-демон (Rust), принимающий задания и запускающий `volter-engine`.
- **DoD:** один run исполняется на отдельной EU-воркер-ноде; e2e «run на удалённом воркере» зелёный.

#### Шаг 4 — Docker overlay + mTLS, реестр нод
- Overlay-сеть, выпуск/ротация сертификатов, bootstrap-токены с TTL.
- Реестр `nodes`: тип (control/worker/stand-test/stand-prod), регион (ru/eu), статус, capacity,
  endpoint, health.
- **DoD:** CP видит здоровье нод; трафик CP↔воркер только по mTLS; graceful-обработка отказа ноды.

#### Шаг 5 — Node-manager: добавление и сетап ноды «по кнопке»
- UI: форма «добавить ноду» (тип, регион, ssh-доступ для bootstrap).
- Сетап по кнопке под тип ноды: набор скриптов + **супервизор прогона** (статус/логи/ретраи) —
  модель из systemd-провижининга «нейроцеха». Создаёт OS-юзера на VPS, ставит Docker, поднимает
  overlay/mTLS, разворачивает образ (worker/stand), регистрирует ноду.
- **DoD:** с чистого EU-VPS нода поднимается одной кнопкой; в UI виден прогресс сетапа по шагам.

### Этап C. Агенты и движок качества

#### Шаг 6 — Враппер авторизации агентов из web
- UI-флоу без LLM: «авторизовать агента на ноде» →
  - **claude:** контейнер → `claude login` → показать ссылку/код в UI;
  - **codex:** `codex auth` → ссылка/код;
  - **aider:** форма API key + URL к API.
- Токены/ключи → secrets-store, привязка к ноде/проекту; монтирование `~/.claude`/`~/.codex` в контейнер.
- **DoD:** claude и codex авторизуются на воркере целиком из web; ключ aider сохраняется и используется.

#### Шаг 7 — Движок качества `volter-engine` (фазы/роли/микроцикл) на Rust
- Реализовать **на Rust** модель SDLC по образцу «нейроцеха», настроенную под README:
  доменные типы → FSM переходов → event-sourcing → роли с границами → микроцикл plan→synth→gate →
  runners → task-loop/poller.
- Связки `agent+model+mode` (chat/planning/execution) и фазовый роутинг моделей по tier с
  fallback-chain (claude→codex→deepseek по остатку лимита) — **без** биллинга.
- **DoD:** диалог «сделай фичу» проходит intake→…→development на воркере через Rust-движок; фазы видны в UI.

#### Шаг 8 — TDD-гейты и «тесты реальные, не mock»
- Реализовать обязательный Definition of Done шага: `test-modeled` → `tests-written` →
  `tests-green` (baseline-diff от merge-base) + **`test-honesty`** + `contract-coverage`.
- Раздельные дорожки backend/frontend, контрактные тесты, e2e на стенде.
- Второй контур — безмодельный артефакт-гейт (`check.cmd`) поверх модельных гейтов.
- **DoD:** шаг не закрывается без написанного, зелёного и честного сьюта; есть тест, что mock-сьют
  отбраковывается `test-honesty`.

#### Шаг 9 — Классификация задач и планирование (author)
- `triage` + `estimate-size` → класс задачи (bug/feature/контент/инфра/research, S/M/L).
- План = базовый SDLC + задаче-специфичные шаги; роутинг ролей/скилов по затронутым компонентам
  (архитектура/UI/вординг/контракты/регуляторика) через манифест-секции и author-перспективы.
- Простой случай (один мета-файл) → короткий rewrite-flow; код → полный flow (скорость↔архитектура).
- **DoD:** для разных классов из одного ввода генерируются разные планы; план редактируемо-читаемый
  до «В работу»; human-in-the-loop ретраи на гейтах.

### Этап D. UI/UX

#### Шаг 10 — Дизайн-система на Tailwind, mobile-first/iPad
- Базовый UI-кит на **Tailwind**: тёмная «приборная» палитра, шрифты, extra-small, компактные контролы.
- Sidebar project-centric (Проект / Задачи / Платформа / Ассистент), адаптация под 320px и iPad.
- **DoD:** ключевые экраны на Tailwind, проходят на 320px и iPad; визуальная регрессия зафиксирована.

#### Шаг 11 — Диалоговый движок: режимы, стрим мыслей, deep-research-прогресс
- **WebSocket/SSE** вместо polling. Переключатель режимов `chat`/`planning`/`execution`
  (+ под-режимы рассуждение/изменение).
- Стрим «мыслей» в простых диалогах («сейчас анализирую…»); прогресс задачи как **deep research**
  (этапы=фазы, под-шаги, живой апдейт).
- Мета-панель: текущая **git-ветка**, стенд, agent+model, статус тестов.
- **DoD:** в одном диалоге переключаются режимы; виден живой прогресс фаз и стрим мыслей.

#### Шаг 12 — Память проекта, индекс, список диалогов
- `.agent/` внутри клона: `index.md` (теги+мета для recall), `dialogs/<id>.md`, `research/<id>.md`,
  `architecture.md` (живая автодока), ignore-файлы (`.aiderignore`/`.codexignore`/`.claudeignore`).
- **Осмысленные авто-названия** диалогов; список как в ChatGPT/Claude с **cost+time**.
- Каждый шаг → коммит в ветку; «зафиксировать» из диалога → коммит диалога+индекса; теги, индекс
  «вектора изменений».
- **DoD:** новый диалог создаёт ветку; после работы в `index.md` появляется запись с тегами; агент в
  следующем диалоге поднимает контекст из индекса.

#### Шаг 13 — Просмотр файлов проекта и аналитика
- Дерево файлов проекта + превью с подсветкой (read-only по умолчанию, README п.41).
- Аналитика: срезы по проекту, по всем проектам, по агенту и **связке agent+model** (cost/time/успех тестов).
- **DoD:** любой файл проекта открывается с подсветкой; дашборд показывает срезы по agent+model.

#### Шаг 14 — Голосовой ввод
- Контрол микрофона → STT → текст агенту (провайдер STT — §18).
- **DoD:** голосовое сообщение распознаётся и обрабатывается агентом в диалоге.

### Этап E. Research, ассистенты, скилы, стоимость

#### Шаг 15 — Research-флоу первого класса
- Сценарий: обсудить → **запустить исследование** (роль researcher, web-discovery) → прогресс как
  deep research → **«сохранить в git» кнопкой** (`.agent/research/<id>.md`) → обсудить → план → «В работу».
- **DoD:** исследование запускается из диалога, результат коммитится в ветку и доступен в плане.

#### Шаг 16 — Project env, DB-ассистент, infra-ассистент
- Project env с доступами (БД и т.д.), секреты в secrets-store (не в git).
- **DB-ассистент**: аналитика/запросы внутри проекта (напр. база tecdoc) через project env.
- **Infra-ассистент**: быстрые правки инфры (nginx и т.п.) в контексте проекта.
- **DoD:** DB-ассистент выполняет аналитический запрос; infra-ассистент правит nginx через диалог.

#### Шаг 17 — Расширяемые skills с генерацией агентом
- Реестр скилов, расширяемый из проекта; путь, когда **агент сам создаёт скил** по запросу
  (генерация + регистрация + тест скила) поверх детерминированной базовой модели.
- **DoD:** по запросу «нужен инструмент X» агент создаёт скил, он регистрируется и используется в фазе.

#### Шаг 18 — Точная стоимость и время (без монетизации)
- Реализовать cost/time-учёт как **проекцию event-лога** (`CostEntry`/`costRollup`): стоимость каждого
  ответа LLM и время его работы, агрегаты по диалогу/проекту/связке agent+model. Цены моделей — из
  конфигурации (прайс per-1M токенов).
- **Не** реализуем: energy/CAP, платежи, пакеты, gateway-биллинг (решение №2).
- **DoD:** для каждого ответа агента видны cost и time; `costRollup` сходится в тестах; платёжных
  сущностей в схеме нет.

### Этап F. Деплой, стенды, релиз

#### Шаг 19 — Деплой проектов на test/prod и e2e на стенде
- Фазы `deploy_stand`/`e2e_stand`/`promotion`: деплой проекта на стенды (compose, rollback, backup,
  nginx wildcard-роутинг), полноценные e2e на test-стенде, prod — только по явному подтверждению.
- **DoD:** проект едет идея→план→разработка→test(e2e зелёные)→prod по кнопке; rollback работает.

#### Шаг 20 — Развёртывание control-plane и сквозная приёмка
- Один скрипт/кнопка для разворачивания **только CP** (RU-VPS); дальше всё из UI (Ш5/Ш6).
- Сквозной приёмочный e2e: поднять CP → добавить EU-воркер кнопкой → авторизовать агента из web →
  добавить git-проект → диалог → research → план → разработка с TDD → test-стенд → prod.
- Финальная автодока, тег релиза.
- **DoD:** сценарий README §62 («поднять инфру, агентов на разных VPS, CP — и работать») проходит
  целиком одним сквозным тестом.
