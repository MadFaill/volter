## 1. Что мы строим (Vision)

Volter — это **единая точка входа в проекты**: умная диалоговая обёртка над любым
coding-агентом (aider / claude / codex), которая управляет распределённой инфраструктурой
(control plane + воркер-ноды + стенды test/prod) и проводит задачу через полный SDLC —
от обсуждения идеи и исследования до плана, TDD-разработки, деплоя и автодокументации.

Ключевые свойства:
- **Диалог как интерфейс.** Всё (общение, исследование, планирование, разработка) — внутри диалога
  с переключаемыми режимами. Минималистичный, mobile-first, под iPad.
- **Агент = абстракция.** aider/claude/codex за единым интерфейсом; связка «агент + модель + mode».
- **Распределённость через Docker.** Control plane на RU-VPS; воркеры (claude/codex) на EU-VPS;
  стенды test/prod — отдельные ноды. Межхостовая связь через Docker.
- **Контекст проекта = склонированный git.** Память, индекс, env, ssh-ключи — на проект.
- **Качество встроено.** TDD обязателен на каждом шаге; задача готова только когда тесты
  написаны, зелёные и доказано реальные.

## 2. Глоссарий

| Термин | Значение |
|---|---|
| **Control Plane (CP)** | Главная нода (RU-VPS) с UI/API/оркестратором. Единственная, что разворачивается вручную. |
| **Worker Node** | EU-VPS, где исполняется агент (claude/codex/aider) в Docker. |
| **Stand (test/prod)** | Нода для деплоя проекта и прогона e2e. |
| **Agent Runtime** | Контейнеризованный процесс конкретного агента + его авторизация. |
| **Binding (связка)** | `agent + model + mode` — выбирается для диалога. |
| **Mode** | Режим связки/диалога: `chat` / `planning` / `execution`. Внутри — под-режимы рассуждения/планирования/изменения. |
| **Project** | Склонированный git-репозиторий + `.agent/` (память, индекс, ignore-файлы, env). |
| **Dialog** | Тред работы. Создаёт git-ветку; шаги коммитятся; сохраняется как md + попадает в индекс памяти. |
| **Skill** | Расширяемый тулинг агента; набор можно дособирать (агент умеет создавать новые). |
| **Task Class** | Классификация задачи, определяющая структуру плана и набор шагов. |

## 3. Топология инфраструктуры

```
                         ┌─────────────────────────────────────────────┐
                         │  CONTROL PLANE  (RU-VPS, ставится вручную)   │
                         │  ┌──────────┐  ┌──────────┐  ┌────────────┐  │
   Пользователь ─────────┤  │  Web UI  │  │   API    │  │ Orchestr.  │  │
   (web / iPad)          │  │ (React)  │──│  (Rust)  │──│  (Rust)    │  │
                         │  └──────────┘  └──────────┘  └─────┬──────┘  │
                         │  Postgres │ Object store │ Secrets │ Node mgr │
                         └───────────────────────────────────┼──────────┘
                                                              │ Docker (mTLS overlay)
                 ┌────────────────────────┬───────────────────┼────────────────────────┐
                 ▼                        ▼                    ▼                        ▼
         ┌──────────────┐        ┌──────────────┐     ┌──────────────┐        ┌──────────────┐
         │ Worker (EU)  │        │ Worker (EU)  │     │  Stand TEST  │        │  Stand PROD  │
         │ claude-cli   │        │ codex-cli /  │     │  деплой+e2e  │        │   деплой     │
         │ runtime      │        │ aider runtime│     │              │        │              │
         └──────────────┘        └──────────────┘     └──────────────┘        └──────────────┘
```

Принципы:
- Разворачивается вручную **только CP**. Остальные ноды добавляются в настройках и
  сетапятся «по кнопке» (см. §10) под свой тип.
- Вся межхостовая связь — через защищённый Docker-транспорт (overlay network + mTLS).
  CP управляет агентами и стендами из контейнера, не по «голому» ssh для рантайма
  (ssh используется только на этапе bootstrap ноды).
- Воркеры stateless по агентам: состояние проекта приезжает как склонированный git +
  `.agent/`; результат уезжает обратно коммитом в ветку.

## 4. Стек

- **Backend / оркестратор:** Rust (axum/tonic для API + gRPC между нодами, tokio).
- **Frontend:** React + TypeScript + **Tailwind CSS**. Mobile-first, адаптация под iPad.
- **Транспорт между нодами:** gRPC поверх mTLS в Docker overlay; стриминг прогресса/мыслей через
  WebSocket (UI) и server-stream gRPC (ноды).
- **Хранилище:** Postgres (метаданные: проекты, диалоги, связки, аналитика), object store
  (артефакты, md-память, исследования), отдельный secrets-store (ssh-ключи, api-ключи, env).
- **Контейнеризация:** Docker / docker compose; образы на каждый тип ноды.
- **Агенты:** claude-cli, codex-cli, aider — в контейнерах с врапперами авторизации.

## 5. Доменная модель и память

### 5.1 Сущности (Postgres)
- `nodes` — тип (cp/worker/stand-test/stand-prod), регион, статус, доступ, токены сетапа.
- `projects` — git-remote, ssh-key-ref, env-ref, путь клона, индекс памяти.
- `bindings` — agent + model + mode + параметры.
- `dialogs` — проект, ветка git, связка, заголовок (осмысленный, автогенерируемый),
  агрегаты cost/time, статус.
- `messages` — роль, контент, режим, cost, time, токены, ссылки на артефакты.
- `plans` — task class, шаги SDLC, статусы шагов, привязка к диалогу.
- `runs` — исполнения шагов на воркерах: cost, time, агент+модель, логи, коммиты.
- `analytics` — срезы по проекту/агенту/связке (cost, time, успех тестов).

### 5.2 Память проекта (`.agent/` внутри клона)
```
.agent/
  index.md            # индекс: теги + мета-контекст по всем диалогам (быстрый recall)
  dialogs/<id>.md     # каждый диалог как markdown (фиксация по кнопке/по шагу)
  research/<id>.md    # результаты исследований (можно «сохранить в git» кнопкой)
  env/                # ссылки/шаблоны env (секреты — НЕ в git, в secrets-store)
  .aiderignore
  .codexignore
  .claudeignore       # ignore-файлы используются при сборке контекста/индекса агентом
  architecture.md     # живая архитектура (автодок)
```
- Контекст проекта = склонированный git. Память — внутри клона, версионируется вместе с кодом.
- Индекс (`index.md`) даёт агенту быстро поднять «о чём речь»: теги, мета, ссылки на диалоги.
- При обращении к агенту контекст достраивается из индекса + ignore-файлов.

### 5.3 Git-workflow (критично)
- **Диалог → создаёт ветку.** Каждый **шаг плана → коммит** в эту ветку.
- В ходе общения можно «зафиксировать» — улетает в ветку (диалог + индекс памяти).
- История размечается **тегами**, формируется индекс «вектора изменений», по которому можно
  проследить эволюцию проекта.
- На **аккаунт/проект — отдельный ssh-ключ** для git (создать ветку, запушить).

## 6. Абстракция агентов и авторизация

### 6.1 Единый интерфейс агента
Трейт `Agent` (Rust) с реализациями `Claude`, `Codex`, `Aider`:
- `start(binding, project_ctx)` → запускает контейнер на воркере.
- `prompt(mode, message, stream)` → стрим ответа + cost/time/токены.
- `apply_changes()`, `commit(branch)`, `run_tests()`.
- `capabilities()` → какие mode/skills поддерживает.

### 6.2 Враппер авторизации (из web UI, без участия LLM)
При подключении ноды:
1. Поднимается docker-контейнер агента.
2. Вызывается процесс авторизации агента:
   - **claude:** `claude login` → UI показывает ссылку/код.
   - **codex:** `codex auth` → UI показывает ссылку/код.
   - **aider:** UI просит **API key + URL к API**.
3. Токены/ключи сохраняются в secrets-store, привязываются к ноде/аккаунту.

UI-флоу строго из web: жмём «авторизовать» → видим ссылку/код или форму ключа.

## 7. Режимы работы и выбор «как делать»

### 7.1 Mode связки/диалога
- `chat` — обсуждение, консультация по проекту, проработка идей.
- `planning` — формирование плана (см. §8).
- `execution` — разработка по плану.

В диалоге режимы **переключаются**; под-режимы: рассуждение / планирование / изменение.

### 7.2 Выбор стратегии по характеру работы (router)
Оркестратор классифицирует задачу и выбирает flow:
- **Один мета-файл** → простое переписывание (rewrite), без тяжёлого flow.
- **Написание кода** → полноценный flow с балансом скорость ↔ правильная архитектура.
- При **планировании** в зависимости от затронутых компонентов подключаются нужные skills
  (архитектура, UI, вординг, тесты, контракты, регуляторика и т.д.).

### 7.3 Трансляция «мыслей»
В простых диалогах (не работа) показывать статус: «сейчас я анализирую / делаю …».
Прогресс выполнения задачи — в стиле **deep research прогресса** (этапы, под-шаги, живой апдейт).

## 8. SDLC, классификация задач и план

### 8.1 Классификация задачи
Перед планом — определение **task class** (напр.: правка контента / фикс инфры / фича бэкенда /
фича фронтенда / fullstack-фича / исследование / рефакторинг). Класс задаёт скелет шагов.

### 8.2 Структура плана
План = **базовый SDLC** + середина из шагов **под конкретную задачу**:
1. Анализ / контекст (поднять память, индекс).
2. Проработка (для planning): архитектура, UI, вординг/тексты, визуал, контракты, регуляторика.
3. Контракты + тесты вперёд (TDD): e2e, контрактные тесты, бэкенд и фронтенд раздельно.
4. Реализация по flow (скорость ↔ архитектура).
5. Прогон тестов + валидация «тесты реальные, не mock».
6. Деплой на test-стенд → полноценные e2e.
7. Фиксация: коммиты по шагам, теги, обновление автодоки и индекса памяти.
8. (по готовности) деплой на prod.

### 8.3 Соответствие плану
- Чёткие проверки: сделанное **максимально соответствует плану**; если не то — доработки.
- **Human-in-the-loop для ретраев** на каждом критичном шаге.
- Задача **готова**, когда: все тесты написаны, зелёные, и пройдена валидация реальности тестов.

### 8.4 Флоу «идея → результат»
Обсудить чего хочу → запустить исследование → результат исследования **сохранить в git кнопкой**
→ обсудить → сформировать план работ → **запустить разработку**.

## 9. Гейты качества (TDD-движок)

Встроенный движок, без которого шаг не считается завершённым:
- На каждом шаге обязателен **test suite (TDD)**.
- Проверка, что сьют **написан**.
- Проверка, что тесты **зелёные**.
- **Валидатор реальности тестов:** эвристики/анализ, что тесты не mock-заглушки
  (проверка реальных ассертов, обращений к контрактам/БД/e2e, мутационные проверки по возможности).
- Раздельные дорожки: **backend** и **frontend** тесты; **контрактные** тесты; **e2e** на стенде.
- **Живая архитектура + автодок:** `architecture.md` и docs обновляются как часть Definition of Done.

## 10. Управление нодами (provisioning)

- В настройках легко **вводить ноды** prod / test / VPS-агентов.
- Под каждый **тип ноды — сетап по кнопке**: набор скриптов + **супервизор** их выполнения и контроля.
- Сетап умеет: создавать новых **юзеров внутри VPS**, ставить Docker, поднимать overlay/mTLS,
  деплоить нужный образ агента/стенда, регистрировать ноду в CP.
- CP несёт набор скриптов и контролирует их прогон (статус, логи, ретраи).
- Bootstrap ноды — по ssh; дальнейший рантайм — через Docker-транспорт.

## 11. Ассистенты внутри проекта

- **DB-ассистент:** внутри проекта собрать аналитику / запросы / что угодно по БД
  (через project env с доступом к БД, напр. анализ базы tecdoc).
- **Infra-ассистент:** быстро фиксить инфру внутри проекта (nginx и т.п.).
- **Skills:** расширяемый набор тулинга; агент может **сам создать** нужный скилл по запросу.
- **Project env:** на проект задаются env с доступами (БД и пр.), секреты — в secrets-store.

## 12. UI/UX

Принципы: **минималистично, но информативно**, всё **extra small** чтобы влезало, контролы простые и
компактные, **mobile-first**, адаптация под **iPad**, **Tailwind CSS**.

Экраны/блоки:
- **Список проектов** + лёгкое добавление проекта (git-remote + ssh-key + env).
- **Список диалогов проекта** (как в ChatGPT/Claude): осмысленные авто-названия, **cost и time** в строке.
- **Диалог:**
  - переключатель режимов (chat/planning/execution), выбор связки (agent+model+mode);
  - стрим ответа + **«мысли»**, прогресс задачи в стиле deep research;
  - мета-панель: **текущая git-ветка** и прочая мета (стенд, агент, статус тестов);
  - **cost и time** по каждому ответу и по диалогу в целом;
  - кнопки: зафиксировать в ветку, сохранить исследование в git, запустить разработку;
  - контрол **микрофона** — голосовые сообщения, агент их обрабатывает (STT).
- **Просмотр файлов проекта:** превью с **подсветкой кода** (просмотр; редактирование — опционально).
- **Аналитика:** по проекту и по всем проектам, в срезах конкретного агента и связок **agent+model**
  (cost, time, успешность тестов).

## 13. Аналитика и стоимость

- Агрегаты по диалогу, по проекту, по всем проектам.
- Срезы по агенту и по связке agent+model.
- Отображается в списках диалогов и на дашборде аналитики.

## 14. Безопасность

- Per-account/per-project **ssh-ключи** для git (изоляция доступа к репозиториям).
- Secrets-store для ssh-ключей, api-ключей агентов, project env (не в git).
- mTLS на межнодовом транспорте; overlay-сеть Docker; bootstrap-токены нод с TTL.
- Изоляция воркеров (контейнеры; отдельные пользователи на VPS).

## 15. Дорожная карта (фазы)

> Каждая фаза сама подчиняется принципу качества: тесты + автодок + фиксация в git.

**Фаза 0 — Фундамент**
- Каркас репо: Rust workspace (api/orchestrator/agent-traits), React+Tailwind фронт.
- Docker-сборка CP; локальный compose (CP + Postgres + 1 worker-заглушка).
- Базовый CI с TDD-гейтом.

**Фаза 1 — Проекты и память**
- Добавление git-проекта, клон, `.agent/` (index/dialogs/ignore/architecture).
- Per-project ssh-ключ; ветка на диалог; коммит по шагу; теги.
- Список проектов/диалогов в UI.

**Фаза 2 — Агенты и связки**
- Абстракция `Agent` + реализации claude/codex/aider.
- Враппер авторизации из web UI (ссылка/код/api-key).
- Выбор связки agent+model+mode для диалога; стрим ответов + cost/time/«мысли».

**Фаза 3 — Диалоговый движок и режимы**
- Режимы chat/planning/execution; router выбора стратегии по характеру работы.
- Голосовой ввод (STT); прогресс в стиле deep research.

**Фаза 4 — Планирование и TDD-движок**
- Классификация задач; генерация плана (SDLC + задаче-специфичные шаги).
- TDD-гейты: написан/зелёный/реальный; раздельные backend/frontend/contract/e2e дорожки.
- Human-in-the-loop ретраи; проверка соответствия плану.

**Фаза 5 — Распределённость и ноды**
- gRPC+mTLS overlay; запуск агентов на EU-воркерах.
- Provisioning нод «по кнопке» (скрипты + супервизор), создание юзеров на VPS.

**Фаза 6 — Деплой и стенды**
- Деплой проекта на test/prod; полноценные e2e на test-стенде.

**Фаза 7 — Исследования, ассистенты, аналитика**
- Research-флоу (запустить → сохранить в git → обсудить → план).
- DB-ассистент, infra-ассистent, расширяемые skills (агент создаёт скиллы).
- Дашборд аналитики по проектам/агентам/связкам.

**Фаза 8 — Полировка UX**
- iPad-адаптация, extra-small контролы, просмотр файлов с подсветкой, мета-панель ветки.
